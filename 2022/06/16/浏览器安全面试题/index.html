浏览器安全面试题

发表于2022-06-16|更新于2022-06-21|面试题

浏览器安全

XSS-跨站脚本攻击

1.是什么

页面被注入了恶意的js代码，获取用户敏感信息Cookie/SessionID

为了和css区分，原名【Cross-Site Scripting】

本质：恶意代码未经过滤，与网站正常的代码混在一起；浏览器无法分辨哪些脚本是可信的，导致恶意脚本被执行

2.出现时候

用户要提交表单
通过网络劫持动态将js代码修改，返回本地

–页面的内容包括用户输入内容，都在固定的容器中以文本的形式展示
–攻击者利用用户输入片段，拼接特殊字符串，形成代码片段
–攻击者通过在目标网站注入脚本，使之在用户的浏览器上运行
【解决】html进行转义

【漏洞总结–XSS注入方法】
1.以script方式注入
2.内联js，拼接的数据突破限制
3.标签中，恶意内容包含引号，突破属性值限制，注入其他属性/标签
4.href/src属性中，包含javascript:等可执行代码

3.XSS攻击分类

  1. 存储型
    攻击者将恶意代码提交到目标网站的数据库中

    用户打开目标网站，网站服务端将恶意代码从数据库中取出，拼接在HTML中返回给浏览器

    用户浏览器解析后执行，混在其中的恶意代码也被执行

    恶意代码窃取用户数据并发送到攻击者的网站，或者冒充用户行为

    【例子】常见于带有用户保存数据的网站功能，如：论坛发帖，商品评论，用户私信

  2. 反射型
    攻击者构造出特殊URL

    用户点击，网站服务器将恶意代码从URL中取出，拼接到HTML中返回给浏览器

【和存储型的区别】存储型的恶意代码存在数据库中，反射型的存在url中

【例子】网站搜索，跳转–诱骗用户点击
3. DOM型
和反射性差不多
【区别】dom型，取出和执行恶意代码由浏览器端完成，属于前端js自身的安全漏洞，其他两种属于服务端的安全漏洞

4.XSS攻击的预防

xss攻击两大要素：

  1. 攻击者提交恶意代码
  2. 浏览器执行恶意代码

【解决】

  1. 输入过滤—不可靠

  2. 防止浏览器执行恶意代码
    –防止HTML中出现注入
    –防止js执行时，执行恶意代码

    【方法一】：预防存储型/反射型
    改成纯前端渲染，把代码和数据分隔开
    对HTML做充分转义

    【方法二】：预防DOM型
    在使用 .innerHTML、.outerHTML、document.write() 时要特别小心，不要把不可信的数据作为 HTML 插到页面上，而应尽量使用 .textContent、.setAttribute() 等。

    如果用 Vue/React 技术栈，并且不使用 v-html/dangerouslySetInnerHTML 功能，就在前端 render 阶段避免 innerHTML、outerHTML 的 XSS 隐患。

    【方法三】：CSP

    禁止加载外域代码，防止复杂的攻击逻辑。

    禁止外域提交，网站被攻击后，用户的数据不会泄露到外域。
    禁止内联脚本执行（规则较严格，目前发现 GitHub 使用）。
    禁止未授权的脚本执行（新特性，Google Map 移动版在使用）。
    合理使用上报可以及时发现 XSS，利于尽快修复问题。

    【方法四】：
    –输入内容长度控制
    –验证码
    –HTTP-only Cookie: 禁止 JavaScript 读取某些敏感 Cookie，攻击者完成 XSS 注入后也无法窃取此 Cookie。

总结

1.避免内敛事件，尽量使用事件绑定.addEventlistener()
2.避免拼接HTML–使用 createElement、setAttribute 之类的方法实现。或者采用比较成熟的渲染框架，如 Vue/React 等。
3.主动检测–手动/自动扫描工具
